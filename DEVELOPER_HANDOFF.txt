4. ARCHITECTURE NOTES
---

DUAL-MODE AUTH (how the code handles both guest + signed-in):
  - Guest: everything saved to localStorage (key: taste.node.profile.v2)
  - Signed-in: Supabase Auth manages JWT session
    - Frontend: @supabase/supabase-js (gotrue) creates/reads auth user
    - Data layer (web/src/data/api.ts) checks _supabaseActive && userId
      → forwards write/read to supabaseApi.ts
      → otherwise falls back to localStorage
  - If VITE_API_URL is set, also routes to backend for:
    /cluster, /recommendations, /profiles/{id}, /venues

STATUS MODEL (updated 2026-06-27):
  Old: want_to_try | visited | favourite | regular
  New: wishlist    | visited | favourite | not_for_me
  Rationale: "regular" conflated with "favourite"; "Want to Try" was pre-visit
             intent not a relationship label; "Not for me" needed for explicit
             negative signal. `not_for_me` is EXCLUDED from clustering entirely.

VENUE DETAIL MODAL (updated 2026-06-27):
  - "Save for later" → instant wishlist (no form)
  - "I've been here" → full form with Favourite / Not for me / Visited
  - Added list/context selector with AI sentiment parsing
  - "Classic" renamed "Timeless" with explicit explanation

DEMO DATA (updated 2026-06-28):
  - 10,000 Tokyo venues in web/src/data/venues.json (4.4 MB)
  - 100,000 Japan venues in src/data/venues.json (49 MB) — TEMP data,
    capped to 2,000 for local SQLite to keep boot fast.
  - 100 synthetic user profiles (Japanese first names + random last initial)
  - 331 realistic feed posts (2-5 per user, cuisine-aware sentiment)
  - Each demo post now carries deterministic like counts (0-499) computed
    from a hash of the post ID. The "For You" feed sorts by likes descending.

BACKEND DUAL-MODE PERSISTENCE:
  - src/db.py checks if SUPABASE_URL + SUPABASE_SERVICE_KEY env vars are set
    - If YES: re-exports all functions from src/supabase_db.py
    - If NO: uses existing SQLite/SQLAlchemy code
  - src/supabase_db.py mirrors db.py API using supabase-py client
    with the SERVICE_ROLE key (bypasses RLS for server ops)
  - IMPORTANT: When importing VENUE_POOL from src/recommendations
    inside db.py or supabase_db.py, always use
      import src.recommendations as _rec
      _rec._ensure_venues_loaded()
    Do NOT use "from src.recommendations import VENUE_POOL" because
    VENUE_POOL is reassigned when lazily loaded, and "from" imports
    bind to the original empty list.

RLS POLICIES ON ALL USER TABLES:
  - profiles: SELECT/INSERT/UPDATE/DELETE only own row
  - contexts: SELECT/INSERT/UPDATE/DELETE only own contexts
  - ranked_items: same
  - feed_posts: SELECT all (public), INSERT/UPDATE/DELETE own
  - follows: same
  - venues: SELECT by everyone (public catalogue)

6. KNOWN ISSUES & WARNINGS
---

1. Backend Python version uncertainty (.python-version + runtime.txt added)
    → .python-version contains "3.12" (correct per Heroku buildpack docs).
    → runtime.txt contains "python-3.12" (deprecated but takes precedence).
    → If the build STILL fails, the buildpack may be running an old version
      that doesn't support either file. Last resort: switch to a Dockerfile
      or pin Python in Pipfile.
    → Buildpack default is Python 3.14; pydantic-core lacks wheels for 3.14.

2. The supabase-py client cannot connect to Postgres directly
   (IPv6/network restricted). Direct psycopg2/pg8000 connections are blocked.
   All DB ops must go through supabase-py REST or be done in SQL Editor.
   NOTE: The /setup-supabase endpoint was previously broken because it tried
   pg8000. It is now fixed to use REST and is read-only.

3. Large venue JSON files committed to git:
   - web/src/data/venues.json  (4.4 MB, ~10 K venues)
   - src/data/venues.json      (49 MB, ~100 K venues)
   Both are temporary data. The backend is now lazily loaded, and the
   frontend code-splits the JSON into a separate chunk. Neither is parsed
   at app start anymore.

4. No CI/CD pipeline is configured. Deploys are manual (`git push bld main`).

5. pre-commit hooks and linting are not enforced in CI (only local tests).

6. The frontend `web/dist/` is tracked in git because Build.io only runs
   Procfile web process (npx serve), not Node build step. This is a known
   Build.io limitation.

7. Local SQLite venue seeding capped at 2,000 (env var overrideable):
   Set TASTE_NODE_MAX_LOCAL_VENUES=10000 to seed more in local dev.

7. FIXES APPLIED (2026-06-28)
---

a. /setup-supabase endpoint (src/main.py)
   Removed broken pg8000 direct Postgres connection.
   Replaced with read-only diagnostic via supabase-py REST.
   Reports which tables exist and which are missing.

b. Backend lazy venue loading (src/recommendations.py)
   VENUE_POOL is no longer loaded at Python import time.
   _ensure_venues_loaded() lazily populates it on first use.
   Reduces startup memory from ~100 MB to near zero.

   Fixed stale-reference bug in src/db.py and src/supabase_db.py:
   "from src.recommendations import VENUE_POOL" bound to the original empty
   list. After _ensure_venues_loaded() reassigned the global, the local
   import still pointed to []. This caused venue seeding to silently insert

   accessing "_rec.VENUE_POOL" after calling _ensure_venues_loaded().

   Changed static import to dynamic import("./venues.json").
   Vite now code-splits the 3.4 MB JSON into a separate chunk.
   Main JS bundle reduced from 4,041 KB to 650 KB.
   Added chunkSizeWarningLimit: 3500 to vite.config.ts.

d. SQLite venue seeding performance (src/db.py)
   Replaced one-by-one INSERT with executemany batch insert.
   Added TASTE_NODE_MAX_LOCAL_VENUES cap (default 2,000) so the first
   /venues request completes in ~5s instead of timing out.

e. All backend tests still pass (50 passed, 1 skipped).
   Frontend tests still pass (65 passed).

f. "For You" feed popularity sort (web/src/data/api.ts, web/src/views/FeedView.tsx)
   - Added `likes` field to Post type and demo data
   - Deterministic like counts computed from post-ID hash (0-499)

11. KNOWN ISSUES & WARNINGS (updated 2026-06-28)
---

1. PostgREST schema cache staleness after DDL
   After running migrations like `005_full_likes_schema.sql` in SQL Editor,
   PostgREST may not know about new columns for ~1 min. The frontend now has
   multi-stage retry (without offending columns, then minimal columns), but
   if ALL queries fail, feed will show demo data (or empty, depending on auth).
   WORKAROUND: After any DDL, wait 60s before testing the frontend,
   or restart PostgREST from Supabase Dashboard.

2. Build.io propagation lag
   After `git push bld main:master`, Build.io edge cache takes ~1-3 minutes
   to serve the new `index.html` and bundles. During this window, old cached
   HTML references deleted bundles → MIME error → white screen. The v3.0.3+
   builds include a reload guard script that detects empty #root after 5s
   and hard-reloads once. If user STILL sees white screen, advise hard reload
   (Cmd+Shift+R / Ctrl+Shift+R).

3. Backend app name mismatch
   `git remote -v` shows `bld-backend` points to `taste-node-api`.
   The deploy command for backend:
     git push bld-backend backend-release:main
   (NOT `git push bld-backend main` — uses `backend-release` branch
   which has the correct Procfile with `web: uvicorn`).
